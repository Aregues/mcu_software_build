# 软件设计文档

## 1. 引言

### 1.1 编写目的

本文档用于描述“智能恒温发酵箱控制系统”嵌入式软件的设计方案，作为软件开发、联调测试、课程验收和后续维护的依据。  
文档重点说明系统的软件总体架构、任务划分、核心控制逻辑、模块接口、状态机、异常处理和代码组织方式，为编码实现和测试验证提供参考。

### 1.2 项目背景

本项目面向小型食品发酵设备，软件运行于 STM32 微控制器平台，负责完成箱体温湿度采集、加热与风扇控制、用户参数设置、OLED 状态显示、蜂鸣器报警以及串口通信等功能。  
设备适用于酸奶、面团等需要稳定温度环境的学习型实验场景，不追求工业级复杂控制，但要求具备较好的实时性、可靠性和可维护性。

## 2. 软件总体设计

### 2.1 软件目标

本软件的设计目标如下：

- 实现箱内温度和湿度的周期采集与显示
- 支持用户设置目标温度、发酵时长和报警阈值
- 根据设定值自动控制加热片、循环风扇和提示蜂鸣器
- 在传感器异常、超温、参数越界等情况下执行保护逻辑
- 支持通过 UART 输出运行状态，便于上位机调试
- 在 64KB Flash 和 20KB RAM 资源约束下稳定运行

### 2.2 软件设计原则

软件设计遵循以下原则：

- 模块化设计，按功能边界拆分软件
- 驱动层与业务层分离，减少硬件耦合
- 状态机统一管理运行模式，降低控制分支混乱
- 关键数据集中管理，便于调试与测试
- 接口简洁清晰，优先使用明确的输入输出参数
- 不使用动态内存分配，保证嵌入式运行稳定性

### 2.3 软件总体架构

软件总体采用分层结构：

- 驱动层：GPIO、I2C、UART、定时器、PWM、Flash
- 硬件抽象层：传感器访问、显示驱动封装、执行器控制封装
- 服务层：参数管理、故障管理、日志记录、通信打包
- 业务逻辑层：状态机、控制算法、调度任务
- 人机交互层：按键处理、菜单显示、蜂鸣器提示

各层之间遵循单向调用关系，上层通过接口调用下层服务，不直接访问底层寄存器。

## 3. 软件模块划分

### 3.1 模块划分原则

模块划分以功能独立、接口清晰、后续便于替换和复用为原则。  
每个模块只负责单一职责，对外暴露少量必要接口，避免跨模块直接访问内部变量。

### 3.2 模块列表

系统主要模块如下：

- 主程序模块：完成初始化和主循环调度
- 时基调度模块：提供 1ms 系统节拍和周期任务标志
- 传感器采集模块：采集并滤波温湿度数据
- 控制算法模块：计算加热和风扇输出策略
- 参数管理模块：维护设定参数和掉电存储
- 显示模块：驱动 OLED 并组织界面内容
- 按键输入模块：处理按键扫描与菜单操作
- 通信模块：提供串口状态上报和参数读写
- 故障检测模块：检测并记录运行异常
- 存储管理模块：管理 Flash 参数区和故障记录区

### 3.3 模块关系说明

- 主程序模块统一初始化所有模块，并在主循环中按任务标志调度执行
- 传感器采集模块向控制算法模块和显示模块提供当前测量值
- 参数管理模块向控制算法、显示和通信模块提供配置数据
- 故障检测模块依赖传感器数据、执行器状态和参数阈值进行判断
- 通信模块可读取系统状态和参数，也可触发参数修改请求
- 显示模块综合系统状态、参数和故障码生成用户界面

## 4. 运行机制设计

### 4.1 软件运行模式

系统采用“前后台 + 中断驱动 + 主循环轮询”的运行方式：

- 定时器中断提供 1ms 时基
- 主循环负责检查任务标志并执行对应任务
- 串口接收采用中断缓存方式
- 关键控制逻辑在固定周期任务中执行

### 4.2 任务调度设计

任务划分如下：

- 周期任务：按键扫描、温湿度采样、控制计算、界面刷新
- 事件触发任务：串口报文解析、参数保存请求
- 中断服务任务：系统时基更新、UART 字节接收
- 后台任务：故障记录写入、通信状态维护

### 4.3 周期任务规划

- 1ms：系统节拍更新、软件定时器计数
- 10ms：按键扫描、蜂鸣器驱动、串口超时计数
- 100ms：温湿度采样、传感器有效性检查
- 200ms：控制逻辑执行、风扇与加热输出更新
- 500ms：OLED 主界面刷新、运行时间更新
- 1s：故障检测、状态上报、参数保存触发检查

### 4.4 实时性要求

- 温度闭环控制周期不大于 200ms
- 按键响应时间不大于 100ms
- 超温保护触发响应时间不大于 300ms
- 通信命令从接收完成到处理响应不大于 100ms
- 中断执行时间应尽量短，避免阻塞主循环

## 5. 数据结构设计

### 5.1 数据结构设计原则

数据结构设计要求命名统一、含义明确、字段完整，优先使用结构体管理同类数据。  
共享状态数据集中定义在 `app_data` 模块中，由各业务模块通过接口读写。

### 5.2 系统状态数据结构

系统状态相关结构体包括：

- 系统状态结构体 `SystemState_t`
- 运行标志结构体 `RunFlag_t`
- 故障标志结构体 `FaultFlag_t`

示例字段如下：

```c
typedef struct
{
    uint8_t work_mode;
    uint8_t run_state;
    uint8_t heater_on;
    uint8_t fan_on;
    uint8_t alarm_on;
    uint32_t run_seconds;
} SystemState_t;
```

### 5.3 业务数据结构

业务处理数据结构包括：

- `SensorData_t`：温度、湿度、采样时间戳、有效位
- `ControlOutput_t`：加热输出、风扇档位、报警输出
- `DisplayData_t`：当前页面、刷新标记、显示缓冲
- `CommFrame_t`：地址、功能码、长度、数据区、CRC

### 5.4 参数数据结构

参数数据结构 `UserParam_t` 包含以下内容：

- 目标温度，默认 38.0 摄氏度，范围 20.0 到 45.0
- 发酵时长，默认 8 小时，范围 1 到 24 小时
- 超温报警阈值，默认 42.0 摄氏度，范围 30.0 到 50.0
- 温控回差，默认 0.8 摄氏度，范围 0.3 到 2.0
- 蜂鸣器开关，默认开启，范围 0 或 1

参数写入 Flash 前需进行范围检查，并附带 CRC16 校验。

## 6. 存储与内存管理设计

### 6.1 Flash 空间分配

Flash 空间规划如下：

- `0x08000000` - `0x0800DFFF`：应用程序区，约 56KB
- `0x0800E000` - `0x0800E7FF`：用户参数区，2KB
- `0x0800E800` - `0x0800EFFF`：校准数据区，2KB
- `0x0800F000` - `0x0800F7FF`：故障记录区，2KB
- `0x0800F800` - `0x0800FFFF`：预留区，2KB

擦写策略如下：

- 参数区采用“比较后写入”策略，避免重复擦写
- 故障记录区采用顺序追加方式保存最近 32 条记录
- 校准区仅在维护模式下允许修改

### 6.2 RAM 空间分配

RAM 空间规划如下：

- 全局状态变量区：约 2KB
- 传感器与控制缓存区：约 1KB
- 串口接收发送缓存区：约 1KB
- OLED 显示缓存区：约 1KB
- 局部静态变量及任务临时区：约 2KB
- 中断与函数调用栈空间：预留约 4KB

### 6.3 数据保存策略

- 用户参数在确认保存或退出设置界面后写入 Flash
- 运行时采样值仅保存在 RAM 中，不做掉电保存
- 故障码和最近一次报警时间按需写入故障记录区
- 系统上电后先加载参数，校验失败则恢复默认值

### 6.4 内存安全设计

- 全部缓冲区使用固定长度数组
- 对串口接收长度、菜单索引和数组访问做边界检查
- 关键接口统一检查空指针
- 禁止使用 `malloc` 和 `free`
- 递归调用在本项目中禁止使用

## 7. 软件状态机设计

### 7.1 状态定义

系统定义以下主要状态：

- 上电初始化状态
- 待机状态
- 预热状态
- 正常运行状态
- 参数设置状态
- 报警状态
- 故障停机状态

### 7.2 状态转换条件

- 上电初始化完成且自检通过后进入待机状态
- 用户按下启动键后由待机进入预热状态
- 温度接近设定值后由预热进入正常运行状态
- 用户按下设置键后进入参数设置状态
- 检测到超温或传感器失效后进入报警或故障停机状态
- 故障解除且用户确认后返回待机状态

### 7.3 状态处理内容

- 上电初始化状态：初始化外设、加载参数、自检硬件，完成后转入待机
- 待机状态：显示当前环境值，允许设置参数和启动设备
- 预热状态：优先打开加热器和低速风扇，使温度快速接近目标值
- 正常运行状态：按回差控制或简化 PID 规则稳定温度
- 参数设置状态：冻结控制输出，处理菜单操作和参数修改
- 报警状态：维持安全输出策略，提示用户存在异常
- 故障停机状态：关闭加热器，保留风扇散热，等待人工干预

## 8. 核心控制逻辑设计

### 8.1 控制目标

核心控制逻辑目标是在有限硬件成本条件下，将箱体温度稳定在用户设定值附近，同时避免温度过冲过大，并在异常情况下优先保证设备安全。

### 8.2 输入量与输出量

- 输入：当前温度、当前湿度、目标温度、发酵剩余时间、按键命令、故障标志
- 输出：加热片开关、风扇开关、蜂鸣器开关、OLED 显示内容、串口上报数据

### 8.3 控制流程说明

1. 周期读取温湿度传感器数据
2. 对采样值执行滑动平均滤波
3. 判断当前状态和故障标志
4. 根据目标温度和回差计算加热输出
5. 根据温度变化趋势决定风扇是否辅助循环
6. 更新显示数据和通信上报缓存

### 8.4 关键策略说明

- 回差控制：低于设定值减回差时启动加热，高于设定值加回差时关闭加热
- 延时启动：上电后 3 秒内不允许进入执行器输出
- 超限保护：温度超过报警阈值立即关闭加热并进入报警
- 故障锁定：传感器连续 5 次采样异常时锁定故障停机
- 自动恢复：仅一般报警可自动恢复，故障停机需人工确认

## 9. 各模块详细设计

### 9.1 主程序模块设计

主程序负责完成系统启动和任务调度，主要流程如下：

- 初始化时钟、GPIO、定时器、I2C、UART、OLED
- 初始化软件模块和全局数据
- 加载参数并执行上电自检
- 进入主循环，按任务标志调用对应处理函数

### 9.2 按键输入模块设计

- 按键定义：设置键、加键、减键、启动停止键
- 扫描周期：10ms
- 消抖方式：连续 3 次检测一致才确认状态变化
- 短按用于切换页面和调整单步参数
- 长按用于快速连续调整或启动保存操作
- 多键同时按下时视为异常输入，不执行功能

### 9.3 显示模块设计

- 主界面显示当前温度、目标温度、剩余时间和运行状态
- 设置界面显示参数名称、当前值和范围提示
- 报警界面优先显示故障码和处理提示
- 刷新周期为 500ms，关键状态切换时立即刷新
- 页面状态由显示模块内部状态机管理

### 9.4 参数管理模块设计

- 参数分类：用户参数、校准参数、系统保留参数
- 默认值在编译期定义于 `config.h`
- 参数修改时先写入 RAM 副本，再统一校验
- 用户确认保存后触发 Flash 写入
- 启动时若发现 CRC 错误则装载默认参数并记录一次参数异常

### 9.5 传感器采集模块设计

- 采样对象：箱内温度与湿度
- 采样周期：100ms
- 滤波方式：5 点滑动平均
- 数据有效性判断：校验 I2C 读写结果、数值范围和变化跳变率
- 异常检测：连续采样失败次数达到阈值后置位传感器故障

### 9.6 控制模块设计

- 控制输入：当前温度、目标温度、回差、运行状态、故障标志
- 控制算法：以回差控制为主，辅以简单超调抑制逻辑
- 输出控制条件：仅在预热和运行状态下允许加热输出
- 安全保护逻辑：故障状态下强制关闭加热，风扇按散热策略运行

### 9.7 通信模块设计

- 通信方式：UART，9600bps，8N1
- 报文格式：设备地址 + 功能码 + 数据长度 + 数据 + CRC16
- 收发流程：接收中断缓存、帧完整性判断、命令分发、回包发送
- 校验方式：CRC16-Modbus
- 超时机制：接收间隔超过 20ms 判定为一帧结束

### 9.8 故障检测模块设计

- 故障检测项：传感器失效、超温、参数异常、风扇异常、Flash 写入失败
- 检测周期：100ms 至 1s 不等，按故障类型分别执行
- 故障判定条件：连续异常计数达到门限后正式置位
- 恢复条件：传感器和超温类故障恢复后需稳定一段时间才清除
- 故障上报方式：OLED 提示、蜂鸣器报警、串口故障帧输出

## 10. 接口设计

### 10.1 模块接口设计原则

模块接口应统一命名、明确输入输出方向，避免在接口中暴露内部数据结构实现细节。  
对共享资源的访问统一通过管理模块完成。

### 10.2 接口列表

```c
void System_Init(void);
void Scheduler_Run(void);
void Key_Scan(void);
void Display_Update(void);
uint8_t Sensor_Read(SensorData_t *data);
void Control_Task(void);
uint8_t Param_Load(UserParam_t *param);
uint8_t Param_Save(const UserParam_t *param);
void Fault_Check(void);
void Comm_Task(void);
```

接口说明：

- `System_Init`：完成系统级初始化
- `Scheduler_Run`：在主循环中调度周期任务
- `Sensor_Read`：读取一次传感器并返回结果状态
- `Param_Save`：校验并保存参数到 Flash
- `Fault_Check`：根据当前运行数据刷新故障标志

### 10.3 共享资源说明

共享资源主要包括：

- 全局系统状态结构体
- 当前传感器数据缓存
- 用户参数结构体
- 串口接收缓冲区
- OLED 显示缓冲区

访问约束如下：

- 中断与主循环共享变量需使用 `volatile`
- 多字节共享数据读取时需避免竞争访问
- 参数修改统一在参数管理模块中完成

## 11. 软件时序说明

### 11.1 上电初始化时序

1. 系统时钟初始化
2. GPIO 与基础外设初始化
3. OLED 启动并显示开机界面
4. 参数加载与 CRC 校验
5. 传感器自检与执行器输出自检
6. 状态机进入待机状态

### 11.2 正常运行时序

1. 周期采样温湿度
2. 滤波并校验采样值
3. 刷新故障状态
4. 执行控制策略并更新输出
5. 刷新显示与剩余时间
6. 打包并发送状态数据

### 11.3 典型业务流程时序

- 启动流程：待机 -> 启动按键 -> 预热 -> 达到设定区间 -> 正常运行
- 参数设置流程：进入设置界面 -> 修改参数 -> 校验 -> 保存 -> 返回待机
- 故障报警流程：检测异常 -> 置位故障码 -> 执行保护输出 -> 显示告警 -> 等待恢复
- 通信交互流程：接收命令帧 -> CRC 校验 -> 命令处理 -> 返回响应帧

## 12. 软件异常处理设计

### 12.1 异常分类

- 传感器通信失败
- 温度测量值跳变异常
- 参数越界或校验失败
- Flash 写入失败
- 串口通信超时
- 风扇控制无反馈
- 箱内超温
- 电源欠压

### 12.2 异常处理原则

- 优先保证加热执行器处于安全状态
- 优先保持系统处于可观测、可恢复状态
- 必要时进入停机保护并要求人工确认
- 保留故障码和发生时间，便于分析问题

### 12.3 各类异常处理逻辑

- 传感器异常：连续失败 5 次后置位故障，关闭加热并提示传感器故障
- 参数异常：加载默认参数，同时记录参数恢复事件
- Flash 失败：禁止继续保存，提示存储异常，但允许设备按当前参数运行
- 通信超时：丢弃不完整帧，不影响主控制逻辑
- 超温异常：立即关闭加热，保持风扇运行，蜂鸣器间歇报警
- 电源异常：限制高功耗输出，并在显示界面提示电源告警
