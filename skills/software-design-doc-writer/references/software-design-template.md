# 软件设计文档

## 1. 引言

### 1.1 编写目的

说明编写本软件设计文档的目的，作为软件开发、测试、验收和维护的依据。  
本文档用于描述系统的软件总体结构、模块划分、控制逻辑、接口关系、状态转换、异常处理及代码组织方式，为后续编码实现提供指导。

### 1.2 项目背景

说明项目名称、应用场景及软件承担的任务。  
例如：本软件用于某温控设备的控制系统，实现温度采集、参数设置、运行控制、状态显示、故障检测与保护等功能。

### 1.3 参考资料

列出设计依据的文档，例如：

- 软件需求规格说明书
- 硬件原理图
- 通信协议说明书
- 芯片数据手册
- 相关行业标准

### 1.4 术语与缩略语

对文档中涉及的专业术语和缩略语进行解释。  
例如：

- MCU：微控制器
- ADC：模数转换
- EEPROM/Flash：非易失性存储器
- UI：用户界面

## 2. 软件总体设计

### 2.1 软件目标

根据需求文档说明软件应实现的目标，包括：

- 功能目标
- 性能目标
- 实时性要求
- 安全性与可靠性要求
- 资源约束要求

### 2.2 软件设计原则

说明软件设计遵循的原则，例如：

- 模块化设计
- 高内聚、低耦合
- 可维护性
- 可扩展性
- 可测试性
- 可实现性

对于嵌入式软件，除上述通用原则外，建议进一步明确：

- 以确定性、可预测时序和最坏情况可分析为优先目标，而不是单纯追求平均性能或峰值吞吐量
- 以系统稳定性、资源边界清晰和故障行为可控为优先目标，避免引入难以验证的运行时复杂性
- 优先采用静态、可审查的设计方式，例如明确任务周期、缓冲区边界、超时处理和恢复条件
- 若性能优化与实时性、任务周期稳定性、中断响应或异常处理一致性冲突，应优先保证后者
- 对动态内存、阻塞等待、重试机制、跨上下文共享数据等可能影响确定性的设计选择，应明确约束条件或说明不采用

可参考写法：

“本系统的软件设计遵循确定性优先原则。在满足功能需求的前提下，优先保证关键控制链路和保护链路的响应时间可预测、任务调度关系清晰、资源占用有界、异常状态可恢复。对于可能带来不确定延迟的设计，如动态内存分配、无界阻塞等待或不可控重试，原则上不采用；如确需采用，必须在设计中说明适用范围、上界和风险控制措施。”

### 2.3 软件总体架构

说明软件整体结构及分层关系，可按下列方式描述：

- 驱动层
- 硬件抽象层
- 服务层
- 业务逻辑层
- 人机交互层

必要时可补充软件架构图，说明各层之间的调用关系和数据流向。

## 4. 运行机制设计

### 4.1 软件运行模式

说明系统采用的运行方式，例如：

- 前后台系统
- 主循环轮询
- 中断驱动
- 事件驱动
- RTOS 多任务调度

建议进一步说明为何选择该运行模式，以及它如何满足嵌入式系统的确定性要求。  
例如：系统采用“定时中断提供时基 + 主循环按标志调度 + 外设中断负责快速采集或通知”的运行模式，以减少不可预测的调度切换开销，并保证关键控制逻辑在固定周期内执行。中断服务程序仅完成必要的快速处理，不在中断中执行耗时业务逻辑。

### 4.2 任务调度设计

说明任务如何分配和执行，例如：

- 周期任务
- 事件触发任务
- 中断服务任务
- 后台处理任务

建议同时写明各类任务的触发源、执行上下文、先后关系及约束边界。  
例如：周期任务由系统时基统一触发，用于传感器采样、控制计算和状态刷新；事件触发任务用于处理按键、通信报文或状态切换请求；中断服务任务仅负责置位标志、采集关键时间点或搬运少量数据；后台任务用于日志整理、非关键状态维护等允许延后完成的操作。对于共享数据，应说明由谁更新、谁读取，以及如何避免竞态或不一致。

### 4.3 周期任务规划

按时间周期列出任务安排，例如：

- 1ms：按键扫描、定时基准更新
- 10ms：传感器采样、故障检测
- 100ms：控制逻辑执行、显示刷新
- 1s：参数存储检查、通信状态维护

除列出周期外，建议补充每类任务的设计依据，例如该周期对应的控制需求、采样特性、通信超时约束或界面刷新要求。  
对于关键任务，可补充说明是否允许抖动、若延迟会带来什么影响，以及是否存在必须优先保障的控制链路或保护链路。

### 4.4 实时性要求

说明关键任务的执行周期、响应时间和优先级要求。

建议优先描述最坏情况要求，而不仅是理想或平均情况。  
例如：

- 温度闭环控制任务执行周期不大于 100ms，最坏情况下响应抖动不超过 20ms
- 超温保护从异常判定成立到关闭加热输出的总响应时间不大于 200ms
- 串口接收中断应在单字节到达后及时完成缓存，不因后台任务阻塞而丢帧
- 非关键显示刷新允许延后，但不得影响控制和保护路径的周期稳定性

如系统采用 RTOS 或多中断源方案，还应说明关键任务优先级关系、抢占原则或延迟控制策略。

## 13. 代码目录建议

建议先由 CubeMX 生成基础工程结构，再新增 `app`、`Module` 文件夹承载业务代码，例如：

```text
Project
├─ Core
│  ├─ Inc
│  └─ Src
├─ Drivers
├─ Middlewares
├─ app
│  ├─ app_init.c
│  ├─ app_task.c
│  └─ app_entry.c
├─ Module
│  ├─ key
│  ├─ display
│  ├─ control
│  ├─ sensor
│  ├─ param
│  ├─ alarm
│  ├─ comm
│  └─ storage
├─ Common
│  ├─ types.h
│  ├─ config.h
│  └─ utils.c
└─ Docs
   └─ software_design.md
```

代码目录设计应体现：

- 保持 CubeMX 自动生成目录的完整性，避免手动改动其内部文件组织
- 用户业务代码集中放在 `app`、`Module`，便于后续重新生成代码时保留
- 模块独立，头文件与源文件成对组织
- 公共配置统一管理
